Bienvenida

Infra Live en breve

Infraestructura declarativa en GCP con Terraform, despliegue de configuración con Ansible y un stack de observabilidad (Prometheus, Grafana, Alertmanager). Acceso seguro vía IAP/OS Login y claves SSH efímeras.

Plataforma

Bootstrap

Proyecto base separado (Bootstrap) que prepara bucket de estado, Service Account runner y Workload Identity Federation (WIF) para usar OIDC desde GitHub Actions sin JSON keys. Seguridad por IDs de repos y rama main.

  • Recursos: bucket tfstate (PAP/UBLA/versioning), SA runner, pool/provider WIF.
  • Condición WIF: restringir por repository_id y refs/heads/main; opcional owner_id.
  • Secrets mínimos: TF_SA_EMAIL y WIF_PROVIDER en el repo Bootstrap.
  • Workflows: Plan (tfplan artifact) y Apply disparado por workflow_run.
# Condición CEL en el provider WIF (ejemplo)
assertion.repository_id == '<REPO_ID>' &&
assertion.ref == 'refs/heads/main'

# Binding a la SA (sin condición)
roles/iam.workloadIdentityUser
roles/iam.serviceAccountTokenCreator

Importante

Pon la lógica en el provider (assertion.*), no en el binding. Protege el bucket tfstate con PAP/UBLA/versioning.

Visión

Arquitectura

  • Infra: Terraform crea red, VM Ubuntu, Cloud NAT y reglas IAP.
  • Acceso: SSH vía IAP + claves efímeras; OS Login opcional.
  • Config: Ansible instala Docker y despliega monitoring + web.
  • Observabilidad: Prometheus + Alertmanager + Grafana tras túnel.

Tip

Mantén el label env=dev en las instancias para que el inventario dinámico las detecte.

Infra

Terraform

Workflows terraform-plan y terraform-apply gestionan cambios. Outputs clave: red, VM, zona, reglas IAP, Cloud NAT.

terraform init -input=false -lockfile=readonly
terraform plan  -input=false -out=tfplan.bin
terraform apply -input=false tfplan.bin

Importante

Incluye tu cuenta/SA en iap_members para abrir túneles IAP. OS Login puede anular claves de proyecto si está activo globalmente.

Pipelines

CI/CD

  • terraform-plan: Genera plan y artifacts.
  • terraform-apply: Aplica en main, publica applied-env.
  • Ansible-Inventory: GCloud + IAP, genera inventario y ejecuta playbook.
  • health-report: Check diario, artifacts solo en WARN/CRIT.

Config

Ansible

Inventario dinámico por label env, clave SSH efímera y túnel IAP. Playbook instala Docker, monitoring stack y la web estática.

ansible-playbook -i ansible/hosts.ini ansible/site.yml

Acceso

El artifact iap-tunnel-<env> publica los comandos de túnel para Prometheus (9090), Grafana (3000) y Nginx (8080), además del SSH vía IAP.

Acceso

Túneles IAP

# Prometheus (9090)
gcloud compute start-iap-tunnel <vm-name> 9090 \
  --project=<project-id> \
  --zone=<zone> \
  --local-host-port=9090

# Grafana (3000)
gcloud compute start-iap-tunnel <vm-name> 3000 \
  --project=<project-id> \
  --zone=<zone> \
  --local-host-port=3000

# Nginx (80)
gcloud compute start-iap-tunnel <vm-name> 80 \
  --project=<project-id> \
  --zone=<zone> \
  --local-host-port=8080

Reemplaza los placeholders por tu VM, proyecto y zona. Para shell interactiva usa: gcloud compute ssh <vm-name> --project=<project-id> --zone=<zone> --tunnel-through-iap.

Observabilidad

Monitoring stack

  • Prometheus: http://localhost:9090
  • Grafana: http://localhost:3000
  • Nginx (web): http://localhost:8080

Alertas

Alertmanager notifica vía SMTP (secrets). Ajusta reglas en prometheus/rules/alerts.yml. La alerta de solo lectura ignora tmpfs/ramfs.

Runbooks

Incidencias comunes

  • IAP 403/65535: Revisa rol IAP en SA y regla de firewall para 22 desde IAP.
  • OS Login: Habilitar API y rol roles/compute.osAdminLogin en el proyecto correcto.
  • Apt sin salida: Asegura Cloud NAT activo si no hay IP pública.
  • Alertas RO fs: Ya filtrado tmpfs/ramfs; si persiste, revisa disco.
